# Roadmap: Banking Payment Forecaster

## Overview

Transform three standalone banking scripts into a unified payment forecaster. Phase 1 wires up the data pipeline (Zoho Calendar events parsed into bill-to-account mappings, balances fetched from Monarch and Mercury). Phase 2 builds the forecast engine on top of that pipeline (projected balances, shortfall detection, summary CLI). Phase 3 adds email alerts and daily automation so the forecast runs unattended and warns when trouble is coming.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

## Phase Details

### Phase 1: Calendar Parsing + Bill Mapping
**Goal**: The system can extract upcoming payments from Zoho Calendar, identify which funding account pays each bill, distinguish fixed vs variable amounts, and fetch current balances from both Monarch Money and Mercury
**Depends on**: Nothing (first phase)
**Requirements**: MAP-01, MAP-02, MAP-03, FCST-04
**Success Criteria** (what must be TRUE):
  1. Running the tool pulls Zoho Calendar events and extracts bill name, amount, and due date from event titles
  2. Each extracted payment is matched to a funding (debit) account using the account identifier in the event notes field
  3. Variable-amount payments (e.g., credit card full balance) are visually flagged as estimates in output
  4. Balances are fetched from Monarch Money for personal accounts AND Mercury for business accounts in a single run (not one or the other)

### Phase 2: Forecast Engine + CLI
**Goal**: User runs a single CLI command to see all upcoming payments within a configurable horizon, per-account projected balances after those payments, and clear warnings when any funding account would go negative
**Depends on**: Phase 1
**Requirements**: FCST-01, FCST-02, FCST-03
**Success Criteria** (what must be TRUE):
  1. User can run `task forecast` (or `python payment_forecast.py`) with `--days N` to control the forecast horizon (defaults to 30 days)
  2. Each funding account shows its current balance minus scheduled outgoing payments, with a clear shortfall flag if the projected balance goes negative
  3. A summary line shows total outgoing payments, total available across all funding accounts, and net position for the chosen horizon
**Plans:** 2 plans

Plans:

- [x] 02-02-PLAN.md — CLI output formatting (grouped + timeline views), ANSI colors, exit codes, Taskfile.yml wiring

### Phase 3: Email Alerts + Daily Automation
**Goal**: The system proactively emails the user when a projected shortfall is detected and can run unattended on a daily schedule, sending the full forecast report
**Depends on**: Phase 2
**Requirements**: ALRT-01, ALRT-02, ALRT-03, ALRT-04
**Success Criteria** (what must be TRUE):
  1. When any funding account's projected balance goes negative, a Gmail alert is sent automatically with the shortfall details
  2. Running with a daily-summary flag emails the complete forecast report to the configured recipient
  3. The same shortfall does not produce duplicate email alerts within a configurable window (default 24 hours)
  4. User can verify email delivery with `--test-alert` without needing a real shortfall condition
**Plans:** 2 plans
