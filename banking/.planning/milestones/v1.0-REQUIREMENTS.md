# Requirements: Banking Payment Forecaster

**Defined:** 2026-05-03
**Core Value:** Know before payday whether you can cover every bill -- see exactly which debit account pays which obligation, and get alerted if anything falls short.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Bill Mapping

- [ ] **MAP-01**: System extracts bill-to-account mapping from Zoho Calendar event notes field (e.g., "1 Ink - 7667" identifies the funding debit account by name/last-4-digits)
- [ ] **MAP-02**: System parses bill name and amount from Zoho Calendar event title (format: "Name - $Amount", e.g., "Quickbooks - $38")
- [ ] **MAP-03**: Variable-amount payments (e.g., credit card full balance, utilities) are flagged as dynamic so forecast shows them distinctly from fixed amounts

### Forecast

- [ ] **FCST-01**: User can run `task forecast` (or `python payment_forecast.py`) with a `--days` arg to set the forecast horizon (default 30)
- [ ] **FCST-02**: System detects shortfalls -- flags when total scheduled debits from a funding account exceed its current available balance within the horizon
- [ ] **FCST-03**: User sees a summary view showing total outgoing payments, total available across funding accounts, and net position for the horizon
- [x] **FCST-04**: Forecast pulls real-time balances from Monarch Money for personal accounts and Xero for business accounts (separate sources, not fallback)

### Alerts

- [ ] **ALRT-01**: System sends Gmail email alert when any funding account's projected balance goes negative within the forecast horizon
- [ ] **ALRT-02**: User can run forecast in daily summary mode (cron-friendly) that emails the full forecast report
- [ ] **ALRT-03**: Alerts are idempotent -- the same shortfall does not generate duplicate email alerts within a configurable window (e.g., 24 hours)
- [ ] **ALRT-04**: User can test alerts with a `--test-alert` flag that sends a sample email without requiring a real shortfall

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### Forecast Enhancements

- **FCST-05**: Per-account projected balance breakdown showing balance after each scheduled payment in chronological order
- **FCST-06**: Intra-day payment waterfall showing order-of-payments within a single day
- **FCST-07**: Business-day adjustment -- shift weekend/holiday payment dates to next business day
- **FCST-08**: Balance staleness warnings showing when each account's data was last synced

### Mapping Enhancements

- **MAP-04**: UID-based Zoho Calendar matching (stable even if event title changes)
- **MAP-05**: Fuzzy title fallback matching when exact prefix match fails
- **MAP-06**: payments.yaml as single source of truth for funding accounts -- all tools (forecast, payments, bill-map) read funding_account from payments.yaml instead of Zoho Calendar notes. Zoho Calendar notes auto-updated to show the display name of the linked account (e.g., "Fund: 360 Checking Recurring (4354)") when funding_account changes in payments.yaml. README updated to reflect that all account configuration (including funding account assignment) is done in payments.yaml only

### Extended

- **EXT-01**: What-if scenarios -- simulate adding/removing payments to see balance impact
- **EXT-02**: Historical tracking -- compare past forecasts to actual outcomes
- **EXT-03**: Category filtering -- view forecast for personal vs business accounts separately
- **EXT-04**: Per-account alert thresholds (custom warning levels per funding account)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Web dashboard or UI | CLI-first tool, no browser interface needed |
| Automatic payment execution | Read-only forecasting, never moves money |
| Real-time push notifications | Email alerts only, no mobile/push |
| Multi-user support | Single user, personal finance tool |
| Database backend | File-based config and caching, consistent with existing approach |
| Credit card dynamic balance lookup | Requires Monarch API investigation; defer to v2 accuracy improvement |
