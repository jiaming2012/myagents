## Critical Issues

### CR-01: Unbound variables in 401 retry path

**File:** `xero_balances.py:218-223`
**Issue:** The 401 retry block at lines 218-224 references `api_client`, `accounting_api`, and `tenant_id`, but these variables are defined inside the `try` block starting at line 181. If the `HTTPError` with status 401 is raised during `identity_api.get_connections()` (line 193) or `ApiClient()` construction (line 182), then `tenant_id` and/or `accounting_api` will be unbound, causing a `NameError` that is caught by the broad `except Exception` at line 225 -- masking the real issue with a misleading "retry after 401 failed" message. If the error occurs during `ApiClient()` construction, `api_client` itself is unbound.
**Fix:**
```python
except requests.exceptions.HTTPError as e:
    if hasattr(e, "response") and e.response is not None and e.response.status_code == 401:
        # Token may have been invalidated -- try one more refresh
        try:
            token = refresh_token(token)
            # Re-initialize SDK from scratch to avoid unbound variable issues
            api_client = ApiClient(
                Configuration(
                    oauth2_token=OAuth2Token(
                        client_id=CLIENT_ID, client_secret=CLIENT_SECRET
                    )
                )
            )
            api_client.set_oauth2_token(token)
            identity_api = IdentityApi(api_client)
            connections = identity_api.get_connections()
            if not connections:
                return {}
            tenant_id = connections[0].tenant_id
            accounting_api = AccountingApi(api_client)
            report = accounting_api.get_report_bank_summary(
                xero_tenant_id=tenant_id
            )
            return _parse_bank_summary_report(report)
        except Exception as retry_err:
            print(
                f"Warning: Xero API retry after 401 failed: {retry_err}",
                file=sys.stderr,
            )
            return {}
```

## Warnings

### WR-01: Payment silently dropped when day_of_month is invalid for next month

**File:** `coverage_report.py:86-89`
**Issue:** When a payment's `day_of_month` does not exist in the current month AND has already passed, the code tries the next month. If the day also doesn't exist in the next month (e.g., `day_of_month=31` when current month is July and next month is August -- actually August has 31, but the pattern is fragile), the `except ValueError: continue` at line 88-89 silently drops the payment from the report. The comment at line 74 says "Use last day of month" but the code does not implement that -- it just tries the raw `dom` in the next month. This is a logic error: payments should fall back to the last day of the month, not be silently omitted.
**Fix:**
```python
import calendar

# Replace lines 71-89 with:
try:
    due = datetime(current_year, current_month, dom)
except ValueError:
    # Day doesn't exist in this month -- use last day of month
    last_day = calendar.monthrange(current_year, current_month)[1]
    due = datetime(current_year, current_month, last_day)

if due < today:
    # Already passed this month, use next month
    next_month = current_month + 1
    next_year = current_year
    if next_month > 12:
        next_month = 1
        next_year += 1
    try:
        due = datetime(next_year, next_month, dom)
    except ValueError:
        last_day = calendar.monthrange(next_year, next_month)[1]
        due = datetime(next_year, next_month, last_day)
```

### WR-02: OAuth token file written without restrictive permissions

**File:** `xero_balances.py:57`
**Issue:** `save_token` writes the OAuth2 token (containing access and refresh tokens) to `.xero_token.json` using `Path.write_text()`, which creates the file with default permissions (typically `0644` -- world-readable). While `.xero_token.json` is in `.gitignore`, the file on disk is readable by any user on the system.
**Fix:**
```python
def save_token(token):
    """Save OAuth2 token to file."""
    import stat
    TOKEN_FILE.write_text(json.dumps(token))
    TOKEN_FILE.chmod(stat.S_IRUSR | stat.S_IWUSR)  # 0600 - owner only
```

### WR-03: Monarch API type field access may raise AttributeError

**File:** `coverage_report.py:119`
**Issue:** `acct.get("type", {}).get("name", "")` assumes the `type` field is a dict. If the Monarch API returns `type` as a string or `None` instead of a dict, the chained `.get("name")` call will raise `AttributeError`. The default value `{}` only applies when the key is absent, not when it is present with a non-dict value.
**Fix:**
```python
acct_type = acct.get("type")
if isinstance(acct_type, dict):
    type_name = acct_type.get("name", "")
else:
    type_name = str(acct_type) if acct_type else ""
```

## Info

### IN-01: Requirements not pinned to specific versions

**File:** `requirements.txt:1-7`
**Issue:** Dependencies (`requests`, `python-dotenv`, `pyyaml`, `monarchmoney`, `xero-python`, `pytest`) are not pinned to specific versions. Only `gql<4` has a version constraint. This can cause non-reproducible installs and unexpected breakage when upstream packages release breaking changes.
**Fix:** Pin all dependencies to known-good versions, e.g., `requests==2.31.0`, `pyyaml==6.0.1`, etc. Alternatively, generate a `requirements.lock` or use `pip freeze` output.

### IN-02: Cache key generation uses MD5

**File:** `zoho_calendar_payments.py:306`
**Issue:** `hashlib.md5` is used for generating cache file names. While not a security concern here (cache keys are not used for authentication or integrity verification), some security scanners will flag MD5 usage. Using `hashlib.sha256` would avoid false positives at negligible cost.
**Fix:** Replace `hashlib.md5(key.encode()).hexdigest()` with `hashlib.sha256(key.encode()).hexdigest()[:32]` (truncated for shorter filenames).
